# Phase 1 Plan 3: FastAPI Backend Scaffold Summary

**One-liner:** FastAPI 0.135.2 backend with async health endpoint, Supabase service role singleton via lifespan hook, Pydantic v2 schemas, and Railway Procfile — tested locally returning 200 on GET /health.

## What Was Built

### FastAPI Service at api/

**api/main.py** — Application entry point using the `asynccontextmanager` lifespan pattern. Calls `init_supabase()` once on startup, then yields. Registers the health router via `app.include_router(health.router)`.

**api/routers/health.py** — Single `GET /health` route returning `{"status": "ok"}`. This is the endpoint Railway polls to determine deployment health. Without a 200 response, Railway marks the deployment unhealthy and does not route traffic.

**api/services/supabase_client.py** — Global `_supabase: Client | None = None` singleton. `init_supabase()` creates the client using `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` from environment. `get_supabase()` returns the initialized client or raises `RuntimeError` if called before initialization. The service role key bypasses all RLS policies — used only server-side in FastAPI, never in browser code.

**api/models/schemas.py** — Pydantic v2 schema stubs for Phase 2: `LeadSubmitRequest` and `LeadSubmitResponse`. Implemented as stubs now so Phase 2 only needs to add logic, not restructure imports.

**api/requirements.txt** — Generated by `pip freeze` after installing into `.venv`. Key versions:
- fastapi==0.135.2
- uvicorn==0.42.0
- pydantic==2.12.5 (v2 confirmed)
- httpx==0.28.1
- supabase==2.28.3
- python-dotenv==1.2.2
- stripe==14.4.1

**api/Procfile** — `web: uvicorn api.main:app --host 0.0.0.0 --port $PORT` — Railway reads this to start the service. The `$PORT` variable is injected by Railway at runtime.

**api/.env.example** — All env var keys with empty values. Safe to commit. Real credentials go only in `api/.env` (gitignored).

**.gitignore** — Created at repo root. Excludes `api/.env`, `api/.venv/`, `__pycache__/`, `.DS_Store`, `node_modules/`, `.next/`.

## Supabase Credentials

`api/.env` has empty `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`. Before Railway deployment:
1. Copy values from Supabase dashboard → Project Settings → API
2. `SUPABASE_URL` = Project URL (format: `https://<ref>.supabase.co`)
3. `SUPABASE_SERVICE_ROLE_KEY` = service_role key (NOT anon key)
4. Set these as Railway environment variables (do not commit `api/.env`)

## Railway Deployment Instructions

1. Connect repo to Railway project
2. Set environment variables in Railway dashboard:
   - `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` (required for startup)
   - `GHL_API_KEY`, `GHL_LOCATION_ID`, `CALLRAIL_API_KEY` (required for Phase 2 webhooks)
   - `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET` (required for Phase 2 billing)
   - `RESEND_API_KEY`, `RESEND_FROM_EMAIL` (required for Phase 2 email)
3. Railway detects `api/Procfile` and starts: `uvicorn api.main:app --host 0.0.0.0 --port $PORT`
4. Railway health check polls `GET /health` — must return 200 before traffic is routed
5. Deployment is live when Railway dashboard shows "Healthy"
