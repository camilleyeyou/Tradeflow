<objective>
Create a new Supabase migration that enables Row Level Security on `public.client_users` and adds a single self-read-only SELECT policy. Today `client_users` has RLS disabled by design (see migration 001 comment) — but a public-schema table with RLS disabled is fully readable AND writable via PostgREST by any anon/authenticated user. Because every other table's RLS policy resolves ownership through a `client_users` subquery, an attacker can insert a `(auth.uid(), victim_client_id)` row and defeat all tenant isolation.

The v1.0 "circular dependency" rationale is wrong: a `USING (user_id = (select auth.uid()))` SELECT policy lets a user read only their own mapping — exactly what the other policies' subqueries need. Admin writes use the service role key, which bypasses RLS, so NO write policies are required.

Purpose: Close SEC-02, the root tenant-isolation vulnerability.
Output: `supabase/migrations/004_enable_rls_client_users.sql`.
</objective>

    --
    -- Corrects the v1.0 decision that left client_users without RLS. A public-schema
    -- table with RLS disabled is fully readable AND writable via PostgREST by any

      on public.client_users for select
      using (user_id = (select auth.uid()));
