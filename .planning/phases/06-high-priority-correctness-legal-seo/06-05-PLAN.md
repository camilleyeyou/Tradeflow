Add bot protection to both public POST endpoints (SPAM-01): a hidden honeypot field that silently succeeds without processing when filled, plus per-IP rate limiting that returns 429 when exceeded.

Output: A lightweight in-memory limiter, honeypot fields on both forms, and guards in both API routes.

  <name>Task 1: In-memory sliding-window rate limiter</name>
  <read_first>apps/web/src/app/api/get-started/route.ts (module conventions), CLAUDE.md (serverless note — limiter is best-effort per instance)</read_first>
  <behavior>
    - rateLimit(key, limit, windowMs) returns { allowed: boolean, remaining: number }.
    - The (limit+1)-th call within windowMs for the same key returns allowed=false.
    - Timestamps older than windowMs are evicted so the window slides.
  </behavior>
  <files>apps/web/src/lib/rate-limit.ts</files>
  <action>
    Create apps/web/src/lib/rate-limit.ts implementing a module-level `Map<string, number[]>` of hit timestamps. Export `function rateLimit(key: string, limit = 5, windowMs = 60_000): { allowed: boolean; remaining: number }`: on each call, get/create the array, drop entries older than `Date.now() - windowMs`, and if `length >= limit` return `{ allowed: false, remaining: 0 }`, else push now and return `{ allowed: true, remaining: limit - length }`. Add a top comment documenting this is best-effort per serverless instance (not a global limiter); acceptable for basic bot deterrence per Phase 6 CONTEXT. Also export a helper `function getClientIp(request: Request): string` that reads `x-forwarded-for` (first IP) then `x-real-ip`, falling back to `'unknown'`.
  </action>

  </verify>
  <done>rate-limit.ts exports rateLimit (sliding window) and getClientIp; tsc passes.</done>
